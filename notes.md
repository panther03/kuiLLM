## original

In this repo I am trying to integrate verified GPU kernels I have written in Kuiper (an extension of F*/Pulse that allows writing verified GPU code     using separation logic, that can extract to CUDA; the repo for it is in ~/work/kuiper-jit) in PyTorch, to run an LLM inference pipeline. Kuiper         kernels, being written in a rich dependently typed functional language, have a high degree of polymorphism in terms of things like element types (f32,  bf16, f64, etc.), tile sizes, and even layout of arrays (row or column major), etc. We can't extract such polymorphic functions directly to CUDA, so w   instead manually specify "I want a GEMM of f32 that works on row major tensors" ahead of time and then have a library of extracted .cu files with a    bunch of implementations. Then, we check in the Python side if the operation fits the specifications of a Kuiper implementation we have available       (__init__.py). This gets quite out of hand when working with a variety of different element types and desired tile sizes, however. I would like for     this PyTorch code to instead JIT extract and compile Kuiper kernels based on the inputs to the operator intercepted by the dispatcher. The extracted    kernel should be cached so that whenever the same operator with the same parameters is invoked, that implementation is used, with no compile overhead.  My rough idea of how this will work is as follows: in the dispatcher code (__torch__dispatch__ function in __init__.py), we look up the current aten    operator in a dictionary of supported kuiper implementations. If we get an implementation back, we call a "supported(args, kwargs)" method on that      value, to see if we can extract a suitable implementation. This is more or less the "_supports_kuiper_..._common" functions we already have defined. I   it is not supported, fall back to the stock PyTorch implementation. If it is, call a method on the implementation that looks up and runs an extracted  version of that function in the cache, otherwise extracting and compiling it if there is a miss in the cache. I imagine each of these implementation    objects would define some sort of template to generate the Fstar code that extracts the correct CUDA, probably in jinja or something. These fstar jinj   templates would look like templated versions of the Klas.<Kernel>.fst{i} files in the kuiper repo. They should be as short as possible, avoid having   proofs in the jinja template. We are generating the instantiations in some of these Klas files via shell scripts already (e.g.                          Klas.GEMM.TensorCore2D.fst.sh), so this is probably your best reference. For how to extract the fstar code to CUDA, you should study the makefiles in   the Kuiper repo. This description here is just one way of going about it, feel free to diverge if you think there is a better approach. The important   requirements are: a) the dispatcher code should be light, so no performance overhead in the case that the cache is hot and we don't have to compile     anything , b) there should not be any non-trivial fstar code that is generated by macro, template, etc. any code that has nontrivial proof obligations  belongs in the main kuiper repo. stick to only direct instantiations of kuiper functions, and if you need to write something more complicated, make a   new Klas module. Please do not hesitate to ask questions.


## makeover 

In this repo I am trying to integrate verified GPU kernels I have written in Kuiper (an extension of F*/Pulse that allows writing verified GPU code using separation logic, that can extract to CUDA; the repo for it is in ~/work/kuiper) in PyTorch, to run an LLM inference pipeline. Kuiper kernels, being written in a rich dependently typed functional language, have a high degree of polymorphism in terms of things like element types (f32, bf16, f64, etc.), tile sizes, and even layout of arrays (row or column major), etc. I take advantage of this whenever I see a Python operator by JIT generating an fstar file with these things instantiated, and then extracting it to CUDA and compiling it on the fly. 

So far, I have `aten` (torch) operators `mm` and a number of elementwise kernels implemented and hooked up to their Kuiper implementations. Can you hook up the following `aten` operators, for which suitable Kuiper kernels already exist: `bmm`, `addmm`, `softmax`, and `sdpa`? Here are some tips:
* Each family of operators (for example all elementwise kernels are one family, `mm` is one family) has a class in `kuipy/kuiops.py` has a jinja template for both the fstar instantiation file and the CUDA file that wraps the kernel call, letting it interface with PyTorch's tensors. `supported()` returns None if the requested operator signature (dtype, shape of operands, kwargs, etc.) is not supported by some parameterization of the template, otherwise returning the parameters needed to construct the template instantiation. `run()` then instantiates, compiles it (through _mod), and runs the operator.
* Please make the `supported()` function as broad as possible, and base it only on the limitations of the polymorphic Kuiper kernel. In particular, do not limit the supported datatypes if it is not necessary. Look at the typeclasses and refinements on the types of the Kuiper kernel as the source of truth: for example, a map kernel has no refinements on `et`, while `gemm` requires a `scalar et`, so it can be an integer, float, etc. Finally, the `TensorCore2D` implementation of GEMM also requires a valid fragment input and accumulator type, so for example the input matrices cannot be float32. The `MmImpl` class shows some examples for this.
* Most kernels in Kuiper have a `Kuiper.Kernel.<kernel>` implementation and a `Klas.<kernel>` to go with it. The klas is intended for static extractions (i.e. fixed datatypes and tile sizes etc.), but sometimes the Klas version implements a generic kernel that is slightly more specialized and will save you from having to duplicate some proof steps. An example is Klas.GEMM.TensorCore2D.Inst, which specializes the layout of the tensors to row major - the default in pytorch, so all templated instantiations will share it - which I have used in my implementation of `mm`. Please use your best judgement here to implement the Kuiops against either the Klas or the Kuiper. Don't import the Klas though if it has fixed element types or tile sizes etc., I do not want to see any python code or templates that branch on element types and select between fixed implementations.
* Naming convention: All fstar files and fstar file templates should be in the namespace `Kuiops`, and be proceeded by the titlecase version of the aten op they implement: `Kuiops.Mm`, etc. The fstar template should always be called `Kuiops.Op.Inst.fst.j2`. Supporting kernels and definitions should go in `Kuiops.Op.fst{i}`. For example, if you need to write another polymorphic wrapper over a kernel that instantiates something like a combinator function, then you can put this in a supporting file. Please keep the template files extremely minimal - this applies for the CUDA templates as well.
* You may find some opportunities to share code when implementing `addmm`, as BlockTiling2D supports the alpha and beta parameters needed for a real GEMM. For example, you can presumably reuse `_bt2d_tile` to generate the tile size parameters for BlockTiling2D. But there are some differences, like TensorCore2D not supporting the alpha and beta parameters. Please modularize and share the code as you find appropriate.
* Add unit tests for the operators you add to `tests/`, but you don't have to follow the existing structure of the ones that are there already. Use the structure that you find is most appropriate. 
* Use `make install-kuiper` before starting to copy over the built kuiper files from the kuiper repo ($KUIPER_HOME should already be set, but if it's not, it's in ~/work/kuiper).
* Please keep code concise, try to follow existing conventions, and do not add excessive comments.
* Do not hesitate to ask any questions.
* Please work on a new git worktree.
